# Mjolnirsoft Design Record

This document describes **what the project currently does and why** — shipped reality, not aspiration. Proposed and in-progress design lives in open GitHub issues ("design candidates"); language only lands here once the issue that introduced it has closed. See *Design Documentation* in `CLAUDE.md` for the methodology.

Organised by subsystem. When a component does not yet exist, it does not yet appear here.

---

## Project foundation & tooling

**What it does.** The repository is a TypeScript/Node project. Tests run on [vitest](https://vitest.dev) via `npm test`; `npm run typecheck` runs `tsc --noEmit`. A GitHub Actions workflow (`.github/workflows/ci.yml`) runs the typecheck and the test suite on every push and pull request. Machine-specific configuration is kept out of source via the gitignored `.local.env` convention. First-run setup is automated by `./setup.sh`, which validates that Node and npm are present, installs dependencies, and creates `.local.env` from the checked-in `.local.env.example` template.

**Why.** The orchestration tooling coordinates Claude Code sessions; Claude Code is itself a Node CLI and the Claude Agent SDK ships for TypeScript, so building in the same ecosystem minimises impedance — and the dominant future host-integration surface (VS Code extensions) is TypeScript regardless. vitest was chosen for native TypeScript execution without a separate compile step. CI runs the suite on every push because the project's testing discipline requires tests to pass in CI before an issue closes; wiring it once makes that enforceable rather than aspirational. `setup.sh` exists so a fresh checkout fails fast with actionable guidance when a dependency is missing, rather than surfacing cryptic downstream errors.

## Orchestration core — shared channel

**What it does.** The headless core (`src/core/`) defines the coordination seam. A `Channel` lets participants join under a unique id in a `Role` — `planner` (the authoritative human/architect) or one of the tool-spawned agent roles `orchestrator`, `executor`, `evaluator` — and exchange typed `Message`s; a participant's handler receives every message sent by *other* participants, and `close()` lets it leave and release resources. The core has no terminal or host dependency. The `Message` shape is deliberately minimal (`from`, `role`, `type`, optional `payload`) — an attributed turn in the shared transcript — and expected to evolve. The sender's `role` is **stamped by the channel** on `send` (alongside `from`), so callers supply neither; it travels *in* the delivered message rather than being looked up at the receiver, because a `FileChannel` recipient is a separate process that can't resolve a remote sender's role locally. Carrying the role is what lets a recipient attribute authority — an `executor` peer must never be mistaken for the authoritative human (`planner`) — which the executor's prompt attribution depends on (#86).

Two `Channel` implementations exist:
- **`InMemoryChannel`** — in-process, synchronous delivery; for single-process use and tests.
- **`FileChannel`** — backs a session with a per-session append-only JSONL log (one `Message` per line). Sending appends a line; the channel polls the file and delivers new lines authored by others. The same file is the transport, the persistence (a durable transcript), and the shared conversation, so participants in separate OS processes that open the same path share one channel. A participant sees only messages appended after it joins; constructing the channel with `replay` instead delivers the existing history first and then streams live — the basis for attaching a window to a running session. Replay delivers a participant its **own** historical messages too — only *live* delivery withholds a participant's own sends (which its UI already rendered) — so a window re-attaching after a reload shows its own prior turns, not a half-conversation missing the architect's side (#126).

**Why.** The architecture treats the channel as the single seam between the orchestration engine and any host (terminal, IDE, CI), the way LSP/MCP separate a server from its clients — which is why the transport could be chosen *after* the core and adapter shapes were known. The file-backed transport was chosen so delivery and persistence are one mechanism: an append-only log carries messages across processes and *is* the durable record a session can be queried about later. A file (rather than a broker daemon) is the smallest thing that delivers both, and it sits behind the `Channel` seam, so a broker can replace it later without touching the core or adapters.

## Sessions — addressing by id

**What it does.** Sessions are addressed by id behind a `SessionBackend` seam (`src/core/`): `open(id, {replay})` returns a `Channel` for that id and `list()` returns the known session ids. The `local` backend — `SessionStore` — maps ids to a `FileChannel` over `.mjolnir/sessions/<id>.jsonl`, validating ids to letters, digits, `_` and `-` so a name can never escape the sessions directory (`..` and path separators are rejected). Which backend is mounted is chosen by a committed `mjolnir.config.json` (`storage.backend`): `loadProjectConfig` reads it and `createSessionStore` maps the id to an implementation, defaulting to `local` when the file is absent and failing fast on an unknown backend. A second backend, `git`, runs the live session on the local file channel but on channel close commits the session's record into a dedicated ref (`refs/mjolnir/sessions`) **out of the working tree** — durable and pushable without ever touching the working tree, the index, HEAD, or any checked-out branch; `list()` then enumerates sessions from the ref (unioned with any local-only ones). It uses git plumbing only (`hash-object` → a temp-index tree → `commit-tree` → `update-ref`, with a fixed record identity so it doesn't depend on the repo's git config) and fails fast when git or a repository is absent. The CLI (`--session`), the orchestrator (`spawnExecutor`), the engage handle (`attachInvitation`), and the VS Code view all address sessions by id; the CLI and orchestrator obtain the backend through the factory rather than constructing one.

**Why.** The file is invisible plumbing — a user communicates with executors through the tool, not by editing transport files — so the durable transcript's *location* should never appear in a CLI flag, a handle, or a picker. Hiding it behind an id gives one naming scheme across every surface. Putting the backend behind a seam selected by config (rather than hardcoding the file) means a project chooses its storage strategy once, in a committed file every agent and headless/CI run reads — distinct from the gitignored, per-machine `.local.env` — and further backends are adapters: `git` is the first, with cloud/others able to follow without touching callers. The `git` backend deliberately keeps the live channel local (git is a durable record store, not a live transport) and stores records out of the working tree so they never pollute a developer's branches. The factory is the single place mapping a backend id to an implementation, and validating the session id at the store boundary keeps path-traversal concerns in one place rather than at every caller.

## CLI host adapter

**What it does.** A thin CLI adapter (`src/cli/`) hosts a session from the terminal. Invoked as `<planner|executor> [id] [--session <id>]` (via `npm run session`), it joins a channel in that role and bridges I/O: each stdin line is sent as a `text` message, and messages received from the channel are written to stdout. With `--session <id>` it opens that session through the `SessionStore` (a shared `FileChannel` behind the scenes), so two processes naming the same session share one channel; without it, an in-process `InMemoryChannel`. Adding `--replay` attaches to an existing session and replays its transcript before streaming live — so the session CLI doubles as an interactive window onto a running session, where the user's typed lines are sent as attributed turns. With `--auto` the process runs an automated executor (see below) that responds to messages programmatically instead of bridging a terminal. At startup the adapter loads `.local.env` into the environment (machine-specific config such as `CLAUDE_BIN`), so values a spawned executor needs are present regardless of how the process was launched. A missing or invalid role is rejected with a usage message and a non-zero exit. The terminal bridging (`hostSession`) is separate from the core and takes an injected `Channel` — selecting the channel needs no adapter change.

**Why.** Adapters are how the host-agnostic design reaches a concrete host: the core never imports terminal APIs — only the adapter does. The CLI runs via Node's native TypeScript execution (`node src/cli/main.ts`, Node ≥ 23.6), avoiding a build step while the project is small. Because Node runs TypeScript in *strip-only* mode, the project enables `erasableSyntaxOnly` so only type-erasable syntax is used (no parameter properties, enums, or namespaces) and `tsc` flags violations before they reach the runtime. If broad Node-version compatibility or distribution later matters, a minimum-version check or a build step is the lever (deferred).

## Orchestrator — executor supervision

**What it does.** `spawnExecutor` (`src/orchestrator/`) launches an executor bound to a session (by id, resolved through the `SessionStore`) and returns a handle that reports lifecycle state (`running` → `exited`), can `stop()` the executor, notifies `onExit` listeners, and exposes the orchestrator's own `Participant` (planner role) on that session's channel. The default launcher spawns the executor CLI as a Node child process in automated (`--auto`) mode (`executor <id> --session <id> --auto`), with stdin piped and left open (so the executor stays alive until stopped) and stdout/stderr inherited; the launcher is injectable so the supervisor is unit-tested without spawning a real process. When the executor exits, the orchestrator's channel participant is closed, but the session log — the durable transcript — persists.

**Why.** This is the first component that makes the tool an *orchestrator* rather than two manually-launched peers: it owns the executor's lifecycle. Process life and record life are deliberately decoupled — stopping an executor ends the process while its transcript survives for later inspection. Messaging is reused unchanged from the session-log channel; this layer adds only spawning and supervision, so the seam keeps the orchestrator ignorant of how messages physically travel.

## Automated executor

**What it does.** `runExecutor` (`src/executor/`) joins a session as an executor and replies to each message it receives from another participant, using a replaceable async `Respond` behavior. The shipped `Respond` is `createClaudeCodeResponder`: it spawns a headless Claude Code agent — `claude -p "<task>" --output-format json --settings <permission policy>` — in a per-executor workspace, and replies with its final result — streaming its intermediate output (reasoning, tool use) to the view as it works (see below). The agent runs on the user's logged-in Claude Code subscription (no API key); the `claude` binary is resolved at spawn time via `resolveClaudeBin()` (`CLAUDE_BIN` if set, else `claude`/`claude.exe` from PATH). A `--settings` policy (`EXECUTOR_PERMISSIONS`) pre-allows what a dev task needs — reads, cwd-scoped edits, and shell commands — so the headless executor isn't blocked by permission prompts it can't surface, while denying a few foot-guns — and **denying claude's native sub-agent tool** (`Agent`), so a spawned agent can't spin up its own ad-hoc sub-agents: those are a heavyweight, opaque token sink that balloons the spawner's context and are redundant with our own `mcp__delegate__*` delegation; a bare deny strips the tool from the agent's context entirely, so it falls back to direct reads or asks upward for context it lacks (#131); it is *soft* confinement (native Windows has no OS sandbox, so the worktree cwd, role instructions, and the developer's branch review are the real boundary — #62). The same `--settings` payload also carries `claudeMdExcludes` (glob the repo's `CLAUDE.md`/`CLAUDE.local.md`), so a spawned agent does **not** load the project `CLAUDE.md` (#121): `claude -p` otherwise walks the directory tree to load it (a worktree inherits the repo-root file), an executor would then auto-load norms regardless of role — the issue-discipline ceremony that cost turns and filed a spurious issue. (`--bare` would also drop `CLAUDE.md`, but it skips OAuth/keychain reads and so breaks the subscription login — it was rejected; `claudeMdExcludes` rides settings and leaves auth untouched.) A spawned agent's project norms now arrive via a **pull-on-demand directive** in its composed instructions (#80): the directive tells every agent to read `AGENTS.md` at the repo root and its role-specific norms file (`docs/agents/<role>.md`) before working, pulling the slice appropriate to its role — the orchestrator reads the bookkeeping protocol, executors read the code-change norms — without the uncontrolled auto-load; the human's own (non-headless) Claude Code session keeps the full `CLAUDE.md`. The policy also sets **`autoMemoryEnabled: false`** (#132): claude's built-in auto-memory keys its store to the agent's cwd — an ephemeral worktree — so any note an agent saved would orphan when the worktree is removed and never be recalled by a later session (which runs in a different worktree). A learning worth keeping belongs *up* with the architect, not in an agent's private file, so the role layer tells agents to surface it in the hand-off rather than self-hoard it (#71); auto-memory is a built-in with no tool to deny, so this `--settings` toggle is the off switch. Each spawned agent also runs on a **per-role model** (`--model`, #119): the mechanical roles (executor, evaluator) default to a cheaper, capable tier (`sonnet`) while the orchestrator inherits the user's own default (it carries the design judgment), each overridable in `settings.json` (`mjolnirsoft.model.<role>`) — so bulk implementation runs cheap while coordination stays top-tier. The lever matters because an autonomous agent is token-dense (a single Opus task measured ~1.44M tokens, ~70% of it orchestration overhead): the cost driver is *turns × model*, addressed by cheaper executors here and by removing ceremony in #121. Executor-role instructions are appended to Claude's own system prompt via `--append-system-prompt` (a soft guardrail, overridable per spawn). They compose the extension's **shared model** (`SHARED_CORE`, #71) — the agent chain (architect → orchestrator → executor → evaluator), the factual/design/permission classification, the escalate-when-unsure bias, the **ask-upward-for-context bias** (don't self-survey the project with ad-hoc sub-agents — the context is usually known above you, so ask up; reserve delegation for real tasks, #131), the descriptive-record rule, the **project-bookkeeping boundary** (the change-tracking/release protocol — filing or closing issues, PRs, the commit-and-close ritual, start-of-session tracking checks — is the orchestrator's to own, in coordination with the architect; executors and evaluators surface what needs tracking in their hand-offs rather than enacting the protocol themselves, #80/#121), a **pull-on-demand project-norms directive** (every spawned agent reads `AGENTS.md` at the repo root and its role-specific norms file before working; the files are ordinary repo files in the agent's worktree, no loader or extension-side injection needed, #80), and **session-log literacy** — a recorded human decision's outcome lives in the `interaction-decision`'s `updatedInput.answers` (question → chosen label), not the offered options or any `(Recommended)` marker, so an agent reading another session's log reads the *pick*, not what was merely offered (#102) — so an executor knows its position and routes design and permission decisions up rather than inventing them — with the **executor role insert** and the executor's **operational guidance** (read widely / write narrowly, cover the change with a test, stay in delegated scope and surface discoveries up, self-review a non-trivial change via an evaluator and skip it for a trivial one (#119), don't commit / hand off, justify changes). The pieces are exported and composable, so when the orchestrator and evaluator become tool-spawned agents they reuse the same core with their own insert. Each message the responder turns into a prompt is prefixed with a **sender-attribution line** (`senderAttribution`): `[Message from <descriptor> (id: <from>)]`, a blank line, then the body. The `planner` role reads as `architect — authoritative` (the *only* authoritative sender); an `orchestrator` reads as `orchestrator — delegating` — a (non-authoritative) distinct descriptor so an executor can tell its task-giving supervisor from a peer/subordinate agent while still routing design/permission decisions *past* it to the architect (#114); every other agent role reads as a plain (non-authoritative) `agent`. So the agent can tell an instruction from the authoritative human apart from a peer/subordinate agent's message — the integrity seam delegation (#85) needs before a spawner ever shares its channel with delegates, and the reason the channel stamps `role` into every message (#86, #71). **Routing to an agent is allowlisted, not denylisted** (#116): `runExecutor` calls the responder only for *conversational* messages — a prompt (`text`), a peer's report (`result`), or a failure to react to (`error`) — gated by the single `deliversToAgent`/`AGENT_PROMPT_TYPES` predicate in `executor-runtime.ts`, *before* a turn is ever queued. The shared channel also carries infrastructure — permission/delegation control, per-turn usage tallies (#116), reasoning digests — for transport and the durable log, and that is excluded *by default*: a newly added infra message type can never reach an agent (and be fed back to it as a turn, looping and burning tokens) without being deliberately added to the allowlist. This replaced a per-responder *denylist* of side-channel types, which was fail-open — a new infra type (the per-turn `usage` message) slipped through it and caused exactly such a feedback loop; the allowlist makes that failure structurally impossible. The responder therefore assumes it only ever receives conversation and keeps no side-channel filter of its own. Across an executor's turns the responder pins one Claude session — `--session-id <uuid>` on the first message, `--resume <uuid>` after — so the executor retains context for an interactive exchange instead of cold-starting each message; the id is assigned up front (exposed as `claudeSessionId`) for the keystone (#40) to record. A turn that fails *before a session is established* rotates the id, because `claude` registers a session id even on a failed run (an auth failure does), so a retry that re-created the same id would hit "Session ID … already in use"; once a session is established the id is kept, so a later-turn failure still resumes without losing context (#90). For **reload resilience** (#126) the pinned id is *derived deterministically from the session name* (`claudeSessionIdFor`) rather than random, so a session the extension host had to tear down resumes the *same* `claude` conversation when re-attached — re-deriving the id needs nothing persisted. A re-attached responder starts already-resumed (`resume`: its first turn `--resume`s); if that conversation doesn't exist — an older session from before deterministic ids, or one interrupted before its first turn ran — the run falls back to creating it fresh on the same worktree and retries the turn, so a resume never dead-ends on "No conversation found". `runExecutor` also **serializes turns**: each message's turn is chained on the previous one, so only one `claude` run is in flight at a time. A message that arrives while a turn is still running — commonly while it's blocked on a permission/question card — is queued and run in order once the current turn settles, rather than starting a second concurrent `claude` that would collide on the pinned session id (`Session ID … already in use`, #100); a failed turn still settles the chain, so one failure can't wedge the queue. The run also **streams** (#108): `claude` is invoked with `--output-format stream-json --verbose --include-partial-messages`, and the responder reads the NDJSON line-by-line (`createStreamReader`, pure and unit-tested), feeding the terminal `result` line to the same `interpretClaudeResult` as before and the raw lines to a **single reasoning-digest assembler** that serves *both* the live view and the durable log from one source (so they can't diverge). The assembler (`reasoning-digest.ts`) coalesces the per-token deltas into a **block-level** trail — whole thinking block(s) and the turn's *interim* narration, each assembled verbatim from its deltas (never one entry per delta), interleaved with the turn's tool-uses, each carrying its **action detail**: the tool's input (the actual query/command) and a *trimmed* head+tail slice of its result (large outputs capped at `MAX_TOOL_RESULT_CHARS` so the log doesn't bloat). The turn's **final answer text is excluded** — a trailing text block is held back and dropped (an interim text block is kept once a later block follows it) — so the digest is *reasoning*, not a duplicate of the answer, which renders in its own result bubble. The assembler (`createReasoningDigestAssembler`) is pure of I/O — fed the raw NDJSON lines via a `createStreamReader` `onLine` tap, tracking the assistant message's open content blocks by index (`message_start` restarts indices; `content_block_stop` finalizes a block in order) and attaching each tool's `tool_result` by `tool_use_id` — so the assembly is unit-tested against captured stream lines with no real `claude`. It emits a fresh **snapshot** through an `onReasoningChange` seam each time an entry finalizes (this drives the live view — see the session-view section) and hands the final digest back through `onDigest` on close; the responder posts that as its **own `reasoning-digest` message, before the `result`** and distinct from it (an empty digest — a turn with no reasoning/tools — is skipped, no log noise). So the executor's reply is *block-level reasoning + result*, both durable: a later log-reader (the Arbitrator #99, an Investigator, the orchestrator reviewing) can post-mortem *why* a turn went the way it did, not just *what* it produced (#102), surviving reload/replay — while the per-token snapshots that drive the live view stay ephemeral. (`parseStreamEvent` is retained for `result` detection and direct stream-shape tests; the live view no longer consumes its per-token `ViewEvent`s.) **On the user's subscription** (`apiKeySource:"none"`, no API key) the headless stream carries text + tool-use but **not** extended-thinking blocks, so in practice the trail shows interim narration + tools; the assembler captures thinking correctly *if it ever appears* (unit-tested, forward-compatible). Enabling thinking headless was declined — the `think`/`ultrathink` keyword triggers don't work in `-p`, and the extra-output token cost isn't judged worth the value. This is host-agnostic: the digest rides the same `Respond`→`runExecutor`→channel path as the result (`Respond` now returns a single message *or an ordered list*, which `runExecutor` sends in order), so every host — the CLI too — gets it, not just the VS Code view. The spawn step is injectable, so the round-trip is unit-tested with a fake and CI never invokes the real `claude`; a trivial `acknowledge` responder also exists for transport-only tests. The session CLI runs an executor via `--auto`, and the orchestrator spawns its executors in `--auto` mode, so spawning an executor, sending it a task, and receiving its Claude Code result is a complete coordination round-trip — observable by attaching a `--replay` window to the same session. When a turn *fails* — the `respond` promise rejects, e.g. the `claude` subprocess exits non-zero (expired/absent auth, a missing binary, any error) — `runExecutor` posts the failure onto the channel as an attributed `error` message (and still writes it to `stderr` for host-log detail). Routing it through the channel seam means it reaches every host and the durable log, and — being a reply like any other — it stops the view's "working" indicator instead of leaving it ticking forever on a wedged session (#89). The failure *detail* is taken from `claude`'s stdout JSON `result` (where it writes the human-readable error — an expired/absent login surfaces `"Not logged in · Please run /login"` with `is_error: true`, while stderr is typically empty) via the pure, tested `interpretClaudeResult`, so the surfaced turn is informative and classifiable rather than blank — taking it from stderr alone left it empty. The VS Code view styles an `error` turn with a theme warning colour, and recognises an auth failure to offer guided re-login and retry (#90 — see the session-view section).

**Why.** Coordination needs the executor to *respond*, not merely echo. Keeping the agent behind a single replaceable `Respond` seam meant the transport, lifecycle, and attach were proven with a stub first, then the real Claude Code agent dropped in behind the same seam without touching them — and the seam is where a different agent implementation could attach later. Spawning the `claude` CLI (rather than embedding the Agent SDK) is what runs the agent on the user's subscription: the SDK is documented for API-key auth, which would be separate billing. `CLAUDE_BIN` exists because a spawned subprocess does not always inherit the PATH an interactive shell has — a binary the user can run in their terminal may still be unreachable from the executor. Replies go only to others' messages, and an agent is routed only conversation (the allowlist above), so neither the round-trip nor an infrastructure message can loop the agent.

## Token usage accounting

**What it does.** Each turn's token usage is captured from the `claude` result line — `extractUsage` reads the `input` / `output` / `cache-read` / `cache-creation` token counts into a `Usage` (USD cost is *not* recorded: the subscription's result carries no `total_cost_usd`, so there is nothing to sum). A per-session `UsageMeter` (`usage-meter.ts`) accumulates each turn's `Usage` **by code** — no agent turn is spent counting — and exposes a running total plus a subscribe seam; a meter created with an `onAdd` hook **rolls up** into its spawner's meter, so an orchestrator's total includes every delegate it spawned, recursively. Usage is **persisted per turn** (#116): as each turn completes the host posts a `usage` message (that turn's delta) onto the session channel, so it lands in the durable JSONL continuously rather than being batched into a single total at session close. A long-lived session — the orchestrator can stay open for weeks — therefore keeps an up-to-date accounting that survives teardown (a reload loses at most the in-flight turn, #126), and each turn's cost is individually recoverable for post-mortem ("which turn thought hardest"). The VS Code header shows the running total live: driven by the meter for an active session, or summed from the persisted per-turn deltas when replaying an ended one.

**Why.** Token usage is a performance dimension alongside the elapsed timer (#76) and the per-role model choice (#119) — together the levers of sustainable spend (#118). Capturing it from the result line by code keeps it free (no turn spent counting) and accurate: the result line is the true figure, where summing the project transcript double-counts under `--include-partial-messages`. Persisting **per turn** rather than one close-time total is what makes the accounting correct for a long-lived component and resilient to teardown, and is what turns the log into something analyzable per operation rather than one opaque lifetime number — the latent property that lets a future pass find the heaviest operations and trim their "think". Per-task tallies and context-growth visibility for the orchestrator build directly on this per-turn foundation and are deferred to the orchestrator work (#58), where a task boundary is a first-class concept.

## Agent-to-agent delegation

**What it does.** `createDelegationManager` (`src/executor/delegation.ts`) is the transport-free primitive by which one agent — the *spawner* — delegates to another — a *delegate* — and sees the delegate's report on its own channel (#88, rung 2 of #85). `spawn(role, openingTask)` derives the delegate's id as `${spawnerId}-${role}-${seq}` (a monotonic sequence keeps it unique and the role keeps the lineage legible in logs), opens a **sub-channel** under that id — the delegate's own session log, isolated from the spawner's conversation — starts a delegate responder there, sends the opening task, and **returns the id immediately without awaiting the reply** (the async `Respond` guarantees the round-trip completes after `spawn` returns, so a delegation never blocks the spawner). Each delegate stands up three seats: the **delegate** (a `runExecutor` responder on the sub-channel, in its own role), a **reporter** (the delegate's seat on the *spawner's* channel), and a **driver** (the spawner's seat on the sub-channel). The driver sends the opening task and listens for the delegate's reply, which it bridges **up** through the reporter — so the channel stamps the report `from: <delegateId>, role`, and #86 `senderAttribution` reads it as a (non-authoritative) **agent** report, never indistinguishable from the architect's instruction. The delegate's full exchange stays on its sub-channel; only the reply crosses up — the driver bridges only **conversational** messages from the report seat, gated by the same `deliversToAgent` allowlist the runtime routes on (#116), so a delegate's reasoning digest (#110) and any other infrastructure it emits persist on its own sub-channel log (for later post-mortem) without being fed up to the spawner as a turn to react to; only its distilled report (the `result`/`error`) crosses. `shutdown(id)` leaves all three seats and closes the sub-channel, releasing the bridge (idempotent for an unknown id). `runExecutor` gained an optional `role` parameter (default `executor`, so existing callers are unchanged) so a delegate joins its sub-channel honestly attributed. How a delegate is wired is injectable via `DelegationDeps.createDelegate(role, id, sub) → { close, reportFrom? }`, defaulting to a plain `runExecutor` delegate driven by the `acknowledge` stub: rung 2 proved the whole seam with that echo delegate and no real `claude`, and a real `claude`-backed delegate dropped in behind the same seam in rung 3. #114 generalized the seam from "a responder" to "wire the whole delegate on its sub-channel" so a delegate can be more than a responder — an executor delegate stands up its *own* full session there (worktree, MCP, a nested host); the optional `reportFrom` names which sub-channel seat carries the delegate's report (default: the delegate id), so the bridge tracks the agent's reply even when its agent runs under a suffixed seat (`${id}-executor`) alongside its own MCP seats.

**Live delegation (rung 3, #93).** Making delegation agent-invokable mirrors the permission layer (#66): an executor's `claude` calls an MCP tool — `mcp__delegate__spawn` / `shutdown`, exposed by a second bundled server (`delegation-mcp-server.ts`) and pre-allowed in the executor policy so a spawn never dead-ends on a prompt — which posts a `delegation-request` onto the session channel (`delegation-protocol.ts`); the host answers from `createDelegationHost`, which owns the manager and validates the requested role is a real `AgentRole` before spawning. The channel is the bridge, exactly as for permission prompts, so the standalone server and the in-host manager meet on the same log with no extra transport; the executor's responder and the view both ignore the control messages, while the delegate's *report* arrives as an ordinary attributed turn. The first delegate role is the **evaluator** (`composeAgentInstructions('evaluator')`, #57): a fresh-eyes, no-stake critic that reviews "the changes or state under review" — phrased generally so the one role serves an executor's diff, an orchestrator's design review, or a contributor's PR (anticipated reuses) — and returns a *distilled finding*, never an edit. The role classifies each finding (#104): **legible** ones (objective — a bug, an omission, something that renders invisibly) it scores cold and flags as actionable; **judgment** ones — which turn on a reader-effect a cold read can miss (does a coyness read as intent or as absence, the failure mode a flat rubric mis-ranks) — it tags and routes up for the human to weigh rather than settling with a cold verdict (the inverse, "I couldn't reconstruct the intent here," is likewise a routed judgment, not a scored defect). It raises these by **reporting up the chain** — the evaluator marks, its spawner escalates — chosen over wiring a delegate for direct mid-evaluation escalation (deferred as overlapping #58's inter-session routing). A spawned delegate runs a real `claude` responder with **the spawner's own worktree** as its workspace, so it reviews what *is* (an evaluator runs `git diff` against the executor's uncommitted work); concurrency and read-only safety are held by protocol and role (the spawner delegates at a quiescent point; the evaluator only critiques), not by machinery. The review trigger is task-scoped, not baked into the executor's standing instructions. Proven end-to-end: an executor spawned an evaluator that cold-read its diff and returned a critique the executor then acted on — the evaluator catching, on its first run, that an HTML comment renders invisibly in Markdown.

**Multi-turn delegation (#111).** Delegation is no longer one-shot. Alongside `spawn`/`shutdown`, a pre-allowed `mcp__delegate__send` tool carries a follow-up to a *live* delegate: the host routes it through the delegate's stored driver onto its sub-channel, where the responder takes it as another turn (it pins a `claude` session and `--resume`s) and its reply bridges up as before; `send` reports back whether a live delegate received it, so the spawner learns when one is already gone. This wires the spawner→delegate direction the one-shot pair lacked, so the two can converse mid-task. Role guidance makes the exchange purposeful and bounded: a delegate **asks upward when blocked** — an operational blocker (a command it can't run, a missing PATH, a file it can't locate) or a needed clarification — by ending its turn with the question rather than degrading the task and burying the limitation in its hand-off; the spawner **answers operationally** (enablement — how to run a thing, a path, an env var) but never steers an evaluator's judgment, so the no-stake stance (#71/#104) survives. Surfaced by #109, where a spawned evaluator could only do a static review because it couldn't run the suite and had no way to ask for the PATH.

**Executor-delegate mode (#114, rung 1 of #58).** `createDelegationHost` provisions a delegate by **role**. A critique role (the evaluator) keeps the shared, no-worktree review mode above — a `claude` responder on the *spawner's own* worktree, so it reviews "what is". An `executor` role instead provisions a **fresh isolated worktree + full executor wiring** (responder, permission/delegation MCP, a nested delegation host, a live reasoning stream) on the delegate's own sub-channel via the same session-provisioning helper the `Start … Session` commands use — so the delegate is a *real, attachable* session (it appears in the session list and the architect can open it on demand; no panel is auto-opened) rather than a transient critique. Its distilled hand-off bridges up to the spawner as the spawner's turn; its reasoning trail and permission cards stay on its own sub-channel (the driver bridges only the agent's report seat, excluding the reasoning digest and the delegate's MCP-seat traffic), so an escalation reaches the architect only when they attach to that delegate's tab. The host's `spawnerRole` is now configurable (default `executor`): an orchestrator host stamps its driver seat `orchestrator`, so an executor delegate reads its opening task attributed to its delegating supervisor (#86). This is what makes the AI orchestrator loop (#58) real: an orchestrator session converses with the architect, spawns one executor delegate for a task, and relays the delegate's hand-off back up. **Orchestrator-side review and integration land in #137**: on a hand-off the orchestrator reviews the delegate's branch against the goal (leaning on the executor's own self-review, spawning an evaluator only on doubt) and either **integrates it — pushing the branch and opening a PR**, the architect's merge being the ratification (#71) — or sends the delegate a follow-up to refine (#111). A **role-specific permission policy** (`permissionPolicyFor`) lets the orchestrator `git push` and run `gh` for this (force-push still denied); executors keep the no-push base, since they only hand off. So the orchestrator integrates **from the distilled hand-off** — judging design fit against the goal it set and reaching into the diff only for a specific unresolved point, never re-reading it wholesale (#142): pulling a full diff into its long-lived context is the heavy, lossy work it delegates review to avoid (an early run's integrate turn hit 2.1M cache-read doing exactly that). The architect's authority stays at the merge.

**Why.** This is the foundation the new agent roles (Evaluator, Investigator) and the AI orchestrator (#58) all stand on — they are each delegates — so it is built and proven *standalone* before #58, a thin layer wholly dependent on it. It is deliberately **async and channel-native** rather than a blocking `delegate→await` call: a delegate is just another participant whose messages trigger the spawner's turns exactly as the architect's do, reusing the existing channel + responder model rather than inventing a new mechanism. The **sub-channel** keeps a delegate's full (often verbose) output out of the spawner's context — only the distilled report crosses up, and the spawner can ask for more on the live sub-channel or read the delegate's log later. Posting the report through a seat joined *as the delegate* (rather than re-tagging it) is what makes attribution structural: authority is legible from the channel's own stamping, so a delegate cannot borrow the architect's authority and subvert "route design up" (#71). Mirroring `PermissionBridge`'s transport-free shape — the caller wires the channels and how a sub-channel opens — keeps it unit-testable over `InMemoryChannel`s with no real `claude`, so the intricate bridge plumbing was proven before any agent-invokable tool or live delegate was layered on (rung 3).

## Structured executor–human interactions

**What it does.** #62's `--settings` policy pre-allows common in-worktree work, but a static allow-list can't be exhaustive — a headless executor that hits something it isn't pre-allowed to do would otherwise dead-end with a prompt no one can answer. An executor can be spawned with `--permission-prompt-tool mcp__perm__approve --mcp-config <generated>` pointing at a permission MCP server (`src/executor/permission-mcp-server.ts`) that the system provides. When Claude wants a tool use that isn't auto-approved, it calls that server's `approve` tool with `{ tool_name, input, tool_use_id }`; the server posts an `interaction-request` onto the executor's session channel and blocks until a matching `interaction-decision` returns, then returns the verdict Claude expects — `{ behavior: "allow", updatedInput }` or `{ behavior: "deny", message }`. The channel is the bridge: it is file-backed and cross-process, so the standalone server and the view meet on the same session log with no new transport. The interaction vocabulary lives in `src/core/interaction.ts` (`interaction-request`/`interaction-decision`, plus `decisionToVerdict`); the request-and-await-decision logic is a transport-free `PermissionBridge` (`src/executor/permission-bridge.ts`) unit-tested over an in-memory channel; the Claude responder ignores interaction messages so it never re-enters Claude on its own mid-run request. **What actually escalates:** in headless `claude -p`, print mode auto-approves most tool use (in-worktree edits, Bash) *without* consulting the prompt tool, and gates boundary-crossing actions — a write outside the worktree or to a protected path — which is what reaches the human. So the escalation surface lines up with the worktree-confinement boundary; #62's `deny` rules remain the hard floor for the dangerous auto-approved cases. **The same path carries clarifying questions:** an executor's `AskUserQuestion` arrives through the identical tool as `{ tool_name: "AskUserQuestion", input: { questions } }`, so the view dispatches on `tool_name` to render its options (single- or multi-select) instead of allow/deny, and the pick returns as the allow's `updatedInput = { questions, answers }` — the shape Claude's tool expects — with no server or protocol change, only a second renderer (#68). **The card also learns from a third choice, "Always":** the host records a learned allow-rule at parent-directory granularity (an "Always" on a write to `C:/x/y.txt` remembers `Write(C:/x/**)`) into a gitignored per-project file (`.mjolnir/executor-permissions.json`). The auto-allow then lives in **our MCP server**, not Claude's `--settings`: on each `approve` call the server derives the request's rule (`learnedRuleFor`) and, if it is in the persisted set (`matchesLearnedRule`), returns `allow` itself — no prompt — and posts an audit line to the transcript; otherwise it escalates as above. Consuming the rule in our layer is deliberate and load-bearing — Claude's `--settings` allow rules **don't** reach an out-of-cwd write (it gates those at its access layer before allow rules are consulted, verified across six rule formats plus `--add-dir`), so a learned `--settings` entry would silently do nothing. An unset project dir disables auto-allow so a misconfigured server fails *toward asking*, and the `deny` floor still wins (Claude enforces it before `approve` is called), so a learned rule can't unlock a denied foot-gun (#70). **A hard worktree-confinement guardrail runs first of all (#101):** because the worktree is nested in the repo, every file exists twice, and a careless allow (or a learned rule) on a write to the *main* copy would pollute the developer's working tree — so before any auto-allow or escalation, `approve` runs `outOfWorktreeWriteDenial` (a pure, tested function in `worktree-confinement.ts`) which auto-denies a gated **write** (`Write`/`Edit`/`MultiEdit`/`NotebookEdit`) whose target resolves outside the executor's worktree, returning a message that tells the agent to edit its own copy. Running it *before* the learned-rule check means no "Always" rule can unlock it; reads and in-worktree writes pass through untouched; `Bash` shell-writes are out of scope (no clean path arg — left to the `deny` floor). Path comparison is lexical via `path.win32` (deterministic on Linux CI, correct on the Windows host), canonicalised and prefix-checked so siblings and `..`-escapes can't slip through. The worktree path reaches the server via `MJOLNIR_WORKTREE_DIR`; an unset value fails *toward asking*. This hardens #62's soft confinement: the executor role still says "write only in your worktree," but the guardrail makes it non-bypassable rather than advisory.

**Why.** Both permission approval and `AskUserQuestion` ride Claude's one `--permission-prompt-tool` contract (verified live against the CLI), so the channel mirrors it with a *single* `interaction-request`/`interaction-decision` envelope dispatched on `tool_name`, rather than a message type per interaction — clarifying questions (`AskUserQuestion`) are the second renderer and cost exactly that: a renderer, not a new protocol. Bridging over the existing file-backed channel reuses the one cross-process transport already in the system, so a process Claude spawns and the editor's panel meet without new wiring; the request/decision pair also leaves a durable, attributable audit trail of every boundary-crossing action and its ruling. Staying on `--permission-prompt-tool` (the CLI's headless stand-in for the Agent SDK `canUseTool`) keeps the executor on the subscription, not the API-key SDK.

## Engaging with an executor

**What it does.** `attachInvitation` (`src/orchestrator/`) turns a spawned executor's handle into what a user needs to engage: the executor id, its session id, and the exact `--session <id> --replay` command to open a window onto that session. The orchestrator surfaces this; the user opens the window when they choose, joins as a planner (co-prompter), sees the conversation so far, and can give feedback or corrections that the executor receives as attributed turns.

**Why.** The user's window is just another participant attaching through the channel, so "opening it" is surfacing an attach handle — robust and cross-platform — not new plumbing. Automatically opening a *richer* surface (a graphical Markdown/Mermaid window) is the job of that surface (tracked separately), not a fragile terminal-spawn here.

## VS Code session view (extension)

**What it does.** A VS Code extension (`extension/`, an npm-workspaces package bundled with esbuild) contributes the command *Mjolnirsoft: Open Session View*. It is a list-or-create front door: it lists the workspace's sessions by id (`SessionStore.list()`) in a quick-pick — no file dialog — with a "+ Start a new executor session" option above them (and, when there are no sessions yet, goes straight to starting one); picking an existing session opens a webview panel attached to it. The extension host joins that session as a `planner` participant through the `SessionStore` (`replay: true`), so the panel replays the transcript and then streams live; each message is rendered host-side to HTML (`markdown-it`, with `mermaid` code fences emitted as `<pre class="mermaid">`) and posted to the webview, where a bundled script runs `mermaid` to draw diagrams. Two renderers differ only in newline handling (#95): **human-composed** content — a `planner` turn typed into the composer, and the multi-line/Markdown content a question card shows for review (the #93 drafted-role-text case) — renders with `breaks: true` (`renderComposed`), so every newline survives as a `<br>` and a multi-line message shows as entered rather than collapsing to one line; **agent** (`executor`/`evaluator`) output keeps default soft-break Markdown semantics, which its prose is authored against. The split is *scoped* (chosen over a global `breaks: true`) so the composer is faithful without forcing ragged hard breaks on agent prose; the raw question text still rides in `data-question` untouched, since that's the answer key the webview reads back. The renderer is keyed on the message's `role`, which encodes the authoritative *seat* not authorship — fine while the only `planner` turn the view renders is its own composed send, but a future automated orchestrator emitting prose as `planner` would want a per-message "composed" flag instead. The panel has a multi-line composer: a send posts the whole text as one `text` message and echoes the sent turn locally (the channel does not deliver a participant its own messages); Enter sends and Shift+Enter inserts a newline. Each turn is colour-coded by its sender (a hue derived from the `from` id) so a multi-participant conversation reads at a glance. A "working" indicator — shown **only when a live executor is attached** — ticks an elapsed timer from send until a reply arrives, so a long turn reads as alive rather than wedged; a view-only panel (a session opened without an executor) instead warns on send that the message is logged but won't be answered. The host owns both decisions (it alone reliably knows whether an executor is attached), so the webview just renders what it's told and the indicator can never falsely claim work is happening (#76). When messages stack up behind an in-flight turn (the executor serializes them, #100), the view shows a **queued** cue (`↳ N message(s) queued…`) so a message typed mid-turn reads as waiting, not ignored; only the executor's *own* reply (matched on its participant id) settles a turn and advances the queue, so a bridged delegate report (#93, e.g. an evaluator's finding) renders without prematurely clearing the indicator. The executor's **live reasoning** streams to the view as it works (#108): the responder pushes **block-level digest snapshots** over an in-process, **off-channel** path (`reasoning-stream.ts`, a per-session pub/sub) to the host, which renders each with the *same* renderer the persisted digest uses (`renderReasoningDigestLive`) and posts the HTML to **replace one reasoning box in place** — so a single `💭 Reasoning` box builds up block-by-block as the turn runs (open), with no token-by-token smear. The box holds thinking (dimmed), interim narration, and tool-uses (each a nested twisty showing input + trimmed result); the turn's **final answer renders only in its own result bubble** below, never duplicated into the box. These snapshots are ephemeral: never on the durable channel/JSONL log, gone on reload. The **durable counterpart** is the per-turn **reasoning digest** (#110, assembled in the host-agnostic runtime — see *Automated executor*): it arrives on the channel as its own `reasoning-digest` message just before the `result`. Live (the executor's own panel) it carries the same entries the snapshots already built, so the host renders it once more **collapsed to settle the box in place** (no discard, no double-show, no view swap) and stops tracking it; the digest message does **not** settle the turn (the `result` that follows it does). On **replay/viewer** panels there was no live box, so the digest renders as its own collapsed box — the trail now survives reload for a later log-reader, where before only the final result did. One renderer for both live and persisted is what makes them identical and keeps the box from ever swapping views (the #108 unification — which also folds in #113: distinct blocks render as separate paragraphs). On the subscription surface the stream emits no thinking blocks, so in practice the box shows narration + tools (see *Automated executor*). Closing the panel leaves the session; the panel is created with `retainContextWhenHidden` so switching editor tabs preserves the rendered conversation and any unsent composer text (a hidden webview is otherwise destroyed and recreated empty, and history replays only once at join). With a Claude Code executor running on the same session, a task typed into the composer reaches the executor and its result renders live — the end-to-end objective, verified in the plugin. Two commands start an in-host agent session, both routing through one factored helper (`provisionSession`, parameterized by role, #114). *Mjolnirsoft: Start Executor Session* prompts for a session name; ***Mjolnirsoft: Open Orchestrator*** opens *the* single orchestrator **directly — no name prompt, at a fixed id** — attaching to it if it's already live and otherwise launching/resuming it (#123, the extension now presenting itself as **"Mjolnirsoft Orchestrator"**). For a **worktree role** (executor, evaluator) `provisionSession` creates a dedicated git worktree on a fresh branch (`mjolnir/work/<id>`, via `WorktreeManager`) **based on the freshest `origin/main`** — `currentRemoteBase` fetches and forks the worktree from the latest *merged* code, so a session never starts on a stale local checkout (the trap that put a prior session on pre-rename code); it falls back to local `HEAD` when there's no remote or the fetch fails (#83). The **orchestrator instead works out of the *main repo* with no worktree** (#123): it coordinates and never implements, so a separate branch is only friction, and by the architect's decision it writes the repo like the architect's own session (its confinement boundary is the repo itself, not a worktree). Either way `provisionSession` then writes the per-session MCP config, opens the live reasoning stream, spawns a Claude Code agent **in-process in the extension host** (composing that role's instructions) with that workspace as its cwd, and stands up the in-host delegation host. The command then opens the panel — so a session is started and driven entirely from the editor, no terminal. The orchestrator panel reuses the exact same view (the unified reasoning box, #110, included): the orchestrator's reasoning streams in it the way an executor's does, and its "working" indicator is labelled by role. The panel surfaces the **model** each agent runs on (#119): a persistent session header (`<session> · <role> · model: <model>`) at the top, and the model on each *response* turn header — so a mixed-model view (the orchestrator's own Opus turns next to a bridged Sonnet executor report) tells you, per response, what produced it. (This replaced a transient toast, which truncated and went unread.) An agent's own turns carry the session's model; a bridged delegate report carries the delegate's, looked up in the live-session registry. The factoring is what lets the **delegation host's executor-delegate mode** (#114) provision an executor delegate from the same code path — the delegate is a real session (its log appears in the front-door list) the architect *attaches to on demand* rather than one auto-opened: each in-host session (commands and every spawned executor delegate) registers its live wiring (agent seat, reasoning stream, role) in a small registry while it runs, so when the architect picks it from the front door the panel attaches to the *live* session (live reasoning, role-apt label, and answerable permission cards) instead of a viewer-only replay. The executor edits the real repo on its branch (never the developer's working tree); on session end the system commits its changes onto the branch and removes the worktree, leaving a branch the developer reviews and merges. A **window reload** is survivable (#126): the in-host agent dies, but its worktree and its `claude` conversation persist, so re-opening the session from the front door **resumes** it rather than opening a dead viewer replay — `provisionSession` reattaches to the existing worktree (the signal that a session was *interrupted*, not ended: a clean close removes the worktree) and resumes the conversation, re-registering it live, with the usage tally seeded from the log so the header continues rather than resetting. The session's role is recovered from its log so the right kind of agent is re-provisioned. (The worktree-less orchestrator has no such interrupted-vs-ended signal — being the singleton it simply *always* resumes its one conversation when re-opened, #123.) Activate prunes stale `git worktree` admin entries a prior teardown left behind, while keeping the worktree directories that back resumable sessions. (Re-establishing the live orchestrator→delegate *bridge*, and auto-reopening the panels that were open before the reload, are tracked in #128; a reload still loses the single in-flight turn — inherent to an in-process agent, since `claude` persists only completed turns.) The view also wires that executor's permission escalation (see *Structured executor–human interactions*): it writes a per-session MCP config pointing Claude's `--permission-prompt-tool` at the bundled server (launched with the host's own Node via `ELECTRON_RUN_AS_NODE`, bridging the session's channel log), and renders the executor's interaction requests as inline cards dispatched on `tool_name`: a permission as an allow/deny card, and an `AskUserQuestion` as its selectable options (with empty-submit guarded). A permission card returns just the verdict; a question card returns the picks, which the host assembles against the stored request into `updatedInput = { questions, answers }` before sending the decision back to the executor. The view also turns an executor **auth failure** into guided recovery (#90): it classifies an `error` turn with a heuristic matcher (`isAuthError`, `src/executor/auth-error.ts` — known credential-failure strings: `401`, `Invalid authentication credentials`, `Not logged in`, `OAuth token has expired`/`revoked`, `Please run /login`, matched case-insensitively) and, on a match, renders a recovery card in place of the plain `error` turn: it names the failure as expired/invalid credentials and offers "Log in again" (opens an integrated terminal **launching `claude` directly** as the terminal process via `shellPath`/`shellArgs` — so a `CLAUDE_BIN` path with spaces works on any default shell, no per-shell quoting — running `auth login`, the only path since the official extension exposes no command API for its OAuth flow) and "Retry" (re-sends the host-held last-sent task to the still-alive executor, whose next spawn reads the refreshed per-machine credentials). A non-match falls back to the plain `error` turn — never silent. Classification is **view-side** so the executor/core stays host-agnostic; the recovery affordance is VS-Code-specific.

**Why.** This is the rich counterpart to the terminal CLI window, built VS Code-first because that is where the work happens. It is a host adapter: the webview is pure presentation and the extension host bridges a `Channel`, so the core is untouched and the panel is just another participant attaching to a session — agnostic to whether a human or the orchestrator started the executor on it. The composer sends a whole message at once because the line-per-message limit is a CLI-adapter artifact, not a property of the session. It is its own workspace package because a VS Code manifest and bundling cannot share the root package — and esbuild here is the project's first build step (the headless code still runs build-free on Node). The webview CSP allows `'unsafe-eval'` (Mermaid uses `new Function`) and `'unsafe-inline'` styles (Mermaid injects SVG styling), scoped to our own content inside VS Code's already-sandboxed webview. The Markdown→HTML logic is unit-tested headlessly; the live attach + compose/send and the spawn-from-view flow are verified manually by launching the extension (F5, or sideloaded into a window already open on the repo). The spawned executor runs *in-process* because the extension host's `process.execPath` is VS Code, not node — spawning a node child is awkward, whereas the executor's own `claude` is a separate process found via `CLAUDE_BIN` (loaded from the workspace `.local.env`). It works in a **git worktree** so it can edit the real repo with hard isolation from the developer's working tree, and the system captures its branch at session end for the developer to review — the *agent* orchestrator that would automate that review/commit is a later tier (milestone #8).

_The objective is met end-to-end: a headless Claude Code executor runs on a named session, and a user opens that session in the VS Code view, sends it a task from the composer, and watches the agent's result render live. The AI-orchestrator loop now stands on top of it (rung 1 of #58): the architect starts an orchestrator session, converses with it (watching it reason live), and the orchestrator spawns one executor delegate in its own isolated worktree and relays the delegate's distilled hand-off back to the architect. The orchestrator now also **reviews and integrates** a delegate's work — pushing its branch and opening a PR for the architect to review and merge, or sending it back to refine (#137); the architect's merge is the ratification. **Project norms are discoverable on demand** (#80): `AGENTS.md` at the repo root carries the global rules every agent follows, with a role-directory linking `docs/agents/orchestrator.md` (bookkeeping protocol) and `docs/agents/executor.md` (code-change norms); a pull-on-demand directive in `SHARED_CORE` tells every spawned agent to read these files before working, so the orchestrator owns and runs the bookkeeping protocol rather than routing it above itself, while executors and evaluators still surface discoveries upward rather than enacting the protocol. `CLAUDE.md` is a thin `@AGENTS.md` redirect for other tools; the `claudeMdExcludes` stripping (#121) is preserved so spawned agents never auto-load it._
